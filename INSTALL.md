# Install / Deployment

This file covers the PC (development) vs Raspberry Pi 5 (deployment) split. For the pipeline
itself — control tags, profiling, benchmark mode — see `README.md`.

## PC vs Pi 5

| Environment | File(s) | Purpose |
|---|---|---|
| PC (Windows/Linux exploration) | `requirements-dev.txt` | Full exploration environment: run the demo, experiment, drive the GUI. |
| Raspberry Pi 5 (deployment) | `requirements-pi.txt` + `apt-packages-pi.txt` | Hand-authored CPU-only inference environment. |
| Raspberry Pi 5 (deployment, post-install) | `requirements-pi-lock.txt` | Generated by `scripts/setup_pi.sh` via `pip freeze` — **the actual, authoritative record** of what installed on the Pi. Prefer this over `requirements-pi.txt` once it exists. |

### Why not the old `requirements.txt`? (deleted 2026-07-20)

The legacy `requirements.txt` pinned `apex==0.9.10dev`, which resolves to the wrong package on
PyPI, and mixed in FastSpeech2/Waveglow *training*-only dependencies (`tensorflow`,
`tensor2tensor`, `pyworld`, `tgt`, `tensorboardX`, `scikit_learn`) that are not needed to run
inference against an already-trained checkpoint and may not even build on aarch64.
`minimal_requirements.txt` was the previous PC-use workaround, identical in content to today's
`requirements-dev.txt`. Both old files were deleted in the 2026-07-20 reorg cleanup
(`docs/context/CHANGELOG.md`, `docs/REORG_PROPOSAL.md` §4) once confirmed fully superseded — if
you ever need the original training-environment pins, recover them from git history rather than
reconstructing them by hand.

## Fresh Pi 5 setup

1. Flash Raspberry Pi OS **64-bit** to the SD card (Raspberry Pi Imager).
2. Enable SSH (Raspberry Pi Imager's advanced options, or `sudo raspi-config` on first boot).
3. From the Pi, clone the repo and run the provisioning script:
   ```
   git clone <repo-url> ~/chatterbox
   cd ~/chatterbox
   ./scripts/setup_pi.sh
   ```
4. `scripts/setup_pi.sh` is idempotent — safe to re-run if a step fails (e.g. a flaky weight
   download); it skips work that's already done and prints a PASS/FAIL summary with the path to
   the generated `requirements-pi-lock.txt`.

The script:
- Guards against running anywhere but Linux/aarch64.
- Installs `apt-packages-pi.txt`.
- Creates (or reuses) a venv at `~/chatterbox/venv`.
- Installs `requirements-pi.txt`.
- Downloads the FastSpeech2, FlauBERT, and HiFi-GAN pretrained weights (links from
  `README.md`) into the paths the code expects (`docs/context/ARCHITECTURE.md` "Weights and
  config locations"). Waveglow is skipped by default (not part of the active pipeline).
- Smoke-tests a CPU torch matmul (fatal if this fails) and, best-effort, one end-to-end sentence
  through `do_tts.py` (non-fatal — a missing microphone/speaker or a slow first run on a fresh Pi
  shouldn't fail provisioning on its own).
- Writes `requirements-pi-lock.txt`.

## Mass deployment: golden image

Once `scripts/setup_pi.sh` has run successfully on one Pi 5 and you've confirmed the demo works,
**image that SD card** rather than re-running provisioning on every unit:
- Raspberry Pi Imager → "Use custom" / its image-export option, or
- `dd` the SD card to a `.img` file from another Linux machine (or another Pi).

Flash that golden `.img` directly onto future units' SD cards. This is the correct way to
mass-deploy identical demonstrators — don't try to prepare an SD card for the Pi from Windows by
hand; do the one real provisioning run on actual Pi 5 hardware and clone from there.

## Future enhancement (not implemented)

`requirements-pi.txt` is loosely pinned by hand. For fully reproducible, hash-locked installs,
`pip-tools`/`pip-compile` would be a natural next step (compile `requirements-pi.txt` into a
hash-pinned `requirements-pi.lock` on Pi hardware) — flagged here as a documented option, not
implemented in this change.
